# Product

## Register

product

## Platform

web

## Users

The primary users are Qiriness team members responsible for replying to customer emails. They need a focused setup surface that helps them prepare the future AI reply agent with the right brand voice, tone, and approved knowledge context before the agent is used in operational workflows.

## Product Purpose

This product is a customer-support operating system for Qiriness. The first dashboard surface will configure the input layer for a future email-reply agent: brand voice, tone, and knowledge articles that can be created manually or initialized from Shopify pages before being optimized by the team.

Success means the team can clearly understand what the agent will know, what tone it should use, and which article/page context is approved for future replies.

## Positioning

Qiriness support can shape an AI reply agent from controlled brand guidance and curated Shopify knowledge, instead of relying on opaque or generic automation.

## Brand Personality

Precise and premium. The interface should feel calm, editorially controlled, and trustworthy without becoming decorative, generic, or heavy.

## Anti-references

The dashboard should not look like a Shopify Admin clone, a generic SaaS dashboard, a colorful analytics dashboard, or a heavy CRM.

## Design Principles

Make setup feel like authorship, not configuration. The user is shaping how the agent thinks and writes, so the interface should prioritize clarity, editing, and confidence over settings-page clutter.

Show source and approval clearly. Articles initialized from Shopify pages must make their origin obvious and preserve the distinction between synced source content and user-optimized guidance.

Keep operational density under control. The dashboard can be compact, but the first setup tab should not feel like a data table or CRM screen.

Treat knowledge as a curated library. Brand voice, policies, FAQs, and Shopify-sourced pages should feel like reviewed working material, not raw sync output.

Protect user trust through restraint. Motion, color, and visual emphasis should support state and hierarchy only.

## Accessibility & Inclusion

Use WCAG AA as the baseline. The interface should support keyboard access, visible focus states, readable contrast, reduced-motion preferences, and clear form labels for all setup controls.
